##  Описание проекта

Данный проект представляет собой исследование способов хранения графовых структур в документно-ориентированной базе данных MongoDB. В качестве предметной области выбрана сеть авиаперелётов между городами Азии.

### Цели и задачи

- Провести сравнительный анализ подходов к хранению графов в MongoDB
- Спроектировать граф авиаперелётов (131 город, ~600 маршрутов)
- Реализовать загрузку данных и запросы на Python
- Провести тестирование производительности MongoDB vs Python-структуры
- Визуализировать граф и реализовать алгоритм Дейкстры

### 1. Клонировать репозиторий

2. Создать и активировать виртуальное окружение

bash

bash
brew services start mongodb-community@8.0

bash
python src/generate_data.py

bash

bash
python src/performance_test.py
9. Визуализировать граф

bash
python src/visualize.py
 Сравнение подходов к хранению графов

Подход	Описание
Список смежности (Adjacency List)	Массив ID соседей в документе вершины
Отдельная коллекция рёбер (Edge Collection)	Вершины и рёбра в разных коллекциях
Встраивание рёбер (Embedded Edges)	Рёбра как вложенные документы
Материализованные пути (Materialized Paths)	Предвычисленные пути в документе
Выбранный подход: Отдельная коллекция рёбер (Edge Collection)

Обоснование выбора

Возможность хранить свойства рёбер (weight, airline)
Отсутствие ограничений на количество рёбер у вершины (лимит 16 МБ не критичен)
Поддержка оператора $graphLookup для рекурсивных запросов
Гибкость при добавлении и обновлении рёбер
 Результаты тестирования производительности

Условия теста: 131 город, ~600 маршрутов, 10 повторений, город — Дубай (ID: 102).

Запрос	MongoDB (сек)	Python без MongoDB (сек)	Отношение
1 уровень (прямые соседи)	0.058762	0.000017	3457x
2 уровень (соседи соседей)	0.876036	0.000178	4921x
Выводы

Для небольших графов (до 1000 узлов) Python-структуры работают быстрее за счёт отсутствия сетевых задержек и сериализации.
С ростом объёма данных преимущества MongoDB становятся заметнее благодаря индексам.
Выявлена N+1 проблема при запросах второго уровня — рекомендуется использовать $graphLookup.
MongoDB рекомендуется для графов объёмом >10 000 узлов или когда данные не помещаются в оперативную память.
